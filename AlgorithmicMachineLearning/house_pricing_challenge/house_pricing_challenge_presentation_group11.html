The easiest way to deal with these features could be to delete them all, however, we chose to keep these features instead of simply deleting them because we believe that they can sometimes have a real added value to the house.

'NA' attributes as described in the documentation have been misgiven for Nulls, we thus have to replace them accoridingly if it's a numerical feature or a categorical feature.

Numerical features:

There are three numerical features where there are NaN values in the train set(LotFrontage, GarageYrBlt, MasVnrArea). We deal with them in two different ways:

  • Replacing the NaN value with a 0: Two of them are surfaces, so it makes sense to put a zero instead of a NaN.

  • Deleting the feature: The feature GarageYrBlt is disturbing, it is a date, so replacing a NaN by a zero will be dangerous. We chose to delete this column. Plus, as we will see in the data visualization part it is very much correlated to the YearBuilt feature, so it doesn't add up much information.

However, we have to deal with all the features (pretending that we don't have access to the test set). For the numerical features, there should not be a problem. We'll replace all by 0 since they are all surfaces, or prices, except for MoSold, YrSold, YearBuilt and YearRemodAdd that should all be filled.

Categorical features:

  • Replacing the NaN values by an attribute 'NA': For the categorical features, we only replaced all NaN values by a value 'NA' as intended in the original dataset.
Categorical features encoding¶

There is two main ways to deal with categorical features. Convert each feature into a numerical feature according to a given scale, or do a simple one hot encoding method.

The one hot encoding method allows to keep all the information of a given categorical feature, for a great price of added columns in the dataset. In order to limit greatly the number of added columns, we chose to replace a categorical feature by one numerical feature when it is possible. Meaning, when there is a coherent scale (Ex, Gd, TA, Fa, Po).

Ordinal categorical features encoding:

These are the features that have a certain order, a certain scale. For example, BsmtQual or BsmtExposure. With this method, we lost some information about each feature. Indeed, by choosing a linear scale, we consider that the two differences (Ex / Gd) and (Gd / TA) are equal, which is probably not be the case. Moreover, each feature can have its distribution, that we could analyze more accurately with the data visualization part.

We made this arbitrary choice for two main reasons:

  • a linear scale seems quite fitting for most features, so we didn't falsify that much the information
  • we greatly reduced the added number of columns: avoided the creation of 67 columns.

Finally, there are some scales where an attribute 'NA' was given. We could have chosen to start the scale at zero, giving no weight to a 'NA' attribute. Instead, we chose to apply an imputation algorithm, that replaces the 'NA' values by values that don't modify the global distribution. The algorithm used is MICE (Multivariate Imputation by Chained Equations).

Nominal categorical features encoding:

These are the features that don't seem to have an order, a scale. For instance, wheter there is no Alley, there is a Gravel Alley or a Paved Alley, we cannot really decide which is better than the other. Thus, we one-hot encoded all these variables.

Sales Price analysis¶

The Sale Price mean is around 180 000$ (assuming it's in the American market). Also, the minimum is 34 900$ and the maximum 755 000$ with a standard deviation of 81 000$ which is quite high. We can say then that the range of houses is large, and our dataset diverse.

As said in the descriptive notebook: "The evaluation method for this challenge is the root mean squared error between the logarithm of the predicted value and the logarithm of the observed sales price. Taking logs ensures that errors in predicting expensive houses and cheap houses will have a similar impact on the overall result."

Indeed, the normal/Gaussian assumption is used because it is the most computationally convenient choice. This is a quadratic minimization problem (we use the RMSE as a metric) and can be solved using pure linear algebra. It this case, normality is then a good assumption because it provides us good properties that will probably work well for regression.

However, it has its drawbacks compared to other distributions. It is very sesnsitive to outliers, we'll have to be careful to remove them, in order to not distort our model.

Let's look at the Sale Price distribution, and the log(Sale Price) Distribution.

1.2. Data pre-processing¶

After cleaning and visualizing the dataset, here is what we did:

  • We took the log of SalePrice (described why in the Data Visualization part).
  • We didn't normalize nor standardize the data. One could argue that data needs to be normalized, however for our case, normalization gave worse results than without normalization in the beginning. We chose then to not normalize nor standardize.
  • We created two new features, RichNeighborhood and PoorNeighborhood/
  • We removed outliers we detected manually.
  • To ensure that we remove all potential outliers, we used a more sophisticated technique that detects anomalies in the dataset dristribution. It is an Isolation Forest, that works similarly to a Random Forest but its aim is to detect anomalies in the multivariate distribution

Here are the dataframes' shapes before and after data preparation:

  • train set (1200, 81)
  • X_train (1187, 248) and y_train (1187, 1)
  • test set (200, 81)
  • X_test (200, 248) and y_test (200, 1)

One could argue that we have too many features and could use some feature elimination and feature reduction. But it seems that except for some very important features, the information about the price is well spread on all features. Also, we decided to use linear regression models with regularization (Lasso and Ridge) later on which are models that deal well with high dimensionality.

2.1 Models Testing¶

Firstly, we decided to try a few models and have a look on how well they perform on the training dataset. In oder to do that, we use a cross-validation strategy to validate more accurately our models. Using KFold and shuffling(cross-validation) and using cross_val_score with the right scoring.

In [0]:
#Validation function
def rmse_cv(model, X, y):
    kf = KFold(shuffle=True, random_state=42).get_n_splits(X)
    rmse = -cross_val_score(model, X, y, scoring="neg_root_mean_squared_error", cv = kf)
    return(rmse)

Conclusion¶

To conclude, we chose a model that is easy to understand, and performs well. A Linear Regression using Lasso Regularization for three main reasons:

  • Simplicity
  • Good performance
  • Dealing easily with high dimensionality

Surely, it is not perfect, especially when the actual prices start getting high, our model tends to overestimate the prices.

However, the dataset is very small, we could argue that with more data our model would be better trained.

Also, it is very interesting to see that our model predicts the Sale Price with a price range of 5%, which is lower than the realtor agents negotiation range.
